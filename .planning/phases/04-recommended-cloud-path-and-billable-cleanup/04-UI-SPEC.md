# Phase 4 - UI Design Contract

> Visual and interaction contract for the RunnerKit CLI cloud provisioning and billable cleanup experience. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope

Phase 4 is a CLI-first interaction phase, not a web frontend phase. This contract locks the human terminal output, JSON parity, prompts, confirmation copy, documentation snippets, and semantic visual hierarchy for:

- `runnerkit up` when no BYO host is provided and the recommended cloud path is offered.
- Cloud provisioning plan output before billable resources are created.
- Cloud readiness/progress, status, logs, and doctor provider facts.
- `runnerkit destroy` plan, confirmation, partial-failure, and verified cleanup states.
- README and `docs/cloud-quickstart.md` cloud quickstart copy.

All implementation must preserve the existing `internal/ui.Renderer` pattern: thin command handlers, human output through renderer lines, JSON output with stable fields and `redactions_applied: true`, and no secret/provider token persistence or display.

---

## Design System

| Property          | Value                                                                                      |
| ----------------- | ------------------------------------------------------------------------------------------ |
| Tool              | Manual CLI design system using `internal/ui.Renderer`; shadcn not applicable               |
| Preset            | not applicable                                                                             |
| Component library | none; use Cobra commands, `ui.Renderer`, `ui.Line`, `workflow.Plan`, and markdown docs     |
| Icon library      | none; use existing renderer glyphs `✓`, `!`, `✗`, `?`, `→`, `•` with ASCII fallbacks       |
| Font              | Terminal default monospace for CLI; documentation examples use fenced code blocks/markdown |

### Source-of-truth patterns

- Human output must use existing renderer primitives: `Step`, `Warning`, `Error`, `Success`, `WarningLine`, `ErrorLine`, `PromptLine`, `Next`, and `Bullet`.
- JSON output must expose the same facts as human output using stable keys and must never rely on color or glyphs for meaning.
- Do not introduce a TUI, React, shadcn, third-party terminal UI toolkit, or decorative animation in Phase 4.
- Prefer plain terminal copy that wraps within the existing renderer width behavior; any long provider ID must appear as a separate bullet or JSON field, not embedded in prose.

## Visual Hierarchy and Interaction States

| Surface                                | Focal point                                                      | Visual hierarchy                                                                                     | Required states                                 |
| -------------------------------------- | ---------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| Interactive `runnerkit up` path choice | The selected setup path for `owner/name`                         | Step title → prompt → two numbered options → next-action hint                                        | BYO option, recommended cloud option, canceled  |
| Cloud provisioning plan                | Estimated billable resources and future destroy command          | Warning/cost caveat first → resource table/bullets → labels snippet → confirmation prompt            | dry-run, confirmation required, confirmed       |
| Cloud readiness/progress               | Current readiness gate                                           | Step title → current gate → provider/SSH/cloud-init/preflight facts                                  | pending, passed, failed with remediation        |
| `status`/`doctor` provider facts       | Health summary and provider source line                          | Health line → repository/runner/machine → sources, with Provider alongside GitHub/SSH/Service/Labels | ready, busy, needs_attention, broken, unknown   |
| `destroy --dry-run` plan               | Billable resources that will be deleted or verified non-billable | Warning heading → artifacts in mutation order → billing-impact caveat → next action                  | dry-run, confirmation required                  |
| `destroy` result                       | Verified cleanup outcome                                         | Success/warning/error line → artifact results → pending checkpoints or state removal                 | complete, partial, blocked, already absent      |
| Cloud quickstart docs                  | Fast successful cloud setup path                                 | Prerequisites → run setup → labels snippet → lifecycle commands → destroy verification → limitations | happy path, cleanup path, troubleshooting links |

Accessibility contract:

- Icon/glyph output must always include text labels such as `OK`, `WARNING`, `ERROR`, or descriptive prose; glyphs are never the only signal.
- All prompts must be answerable from text alone in non-color terminals.
- JSON output must include structured equivalents for every human-visible status, warning, cost estimate, resource ID, and pending checkpoint.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage                                                                          |
| ----- | ----- | ------------------------------------------------------------------------------ |
| xs    | 4px   | Inline symbol/text gap; one terminal prefix gap after glyph or ASCII label     |
| sm    | 8px   | Compact bullet grouping; one blank line between short command examples in docs |
| md    | 16px  | Default separation between step title, resource groups, and confirmation copy  |
| lg    | 24px  | Section padding in markdown docs; space before destructive warning blocks      |
| xl    | 32px  | Major quickstart section breaks and command lifecycle groups                   |
| 2xl   | 48px  | Separation between setup and cleanup sections in generated docs                |
| 3xl   | 64px  | Page-level spacing only in long-form docs, not normal CLI output               |

Exceptions: terminal tables and markdown pipes use character alignment rather than pixel spacing; this is allowed only for provider resource tables, requirement traceability tables, and docs tables where column alignment is the visual goal.

## Typography

| Role    | Size | Weight | Line Height |
| ------- | ---- | ------ | ----------- |
| Body    | 16px | 400    | 1.5         |
| Label   | 14px | 400    | 1.4         |
| Heading | 20px | 600    | 1.25        |
| Display | 28px | 600    | 1.2         |

Typography rules:

- Use exactly these four roles for docs and any future rendered UI; terminal output maps them semantically to plain text headings, labels, and bullets.
- Use only two weights: regular `400` and semibold `600`.
- CLI command names, flags, env vars, provider IDs, labels, and resource IDs must be wrapped in backticks in docs and should appear as plain unstyled literals in terminal output.
- Avoid all-caps except stable status words already used by the renderer (`OK`, `WARNING`, `ERROR`, `NEXT`) and environment variable names such as `HCLOUD_TOKEN`.

## Color

| Role            | Value   | Usage                                                                                                     |
| --------------- | ------- | --------------------------------------------------------------------------------------------------------- |
| Dominant (60%)  | #111827 | Primary text and high-contrast terminal/docs foreground intent                                            |
| Secondary (30%) | #F3F4F6 | Markdown code blocks, tables, secondary surfaces, and muted docs backgrounds                              |
| Accent (10%)    | #2563EB | Primary CTA text in docs, active setup-path selection, focus ring if a richer prompt UI is added          |
| Destructive     | #DC2626 | `runnerkit destroy` warnings, billable-resource removal confirmations, unrecovered partial cleanup errors |

Accent reserved for: primary cloud setup CTA (`Provision cloud runner`), active setup-path selection, exact `runs-on` snippet callout, and focus ring in future richer prompts. Accent must not be used for all interactive elements.

Semantic terminal fallback:

- Success: existing `✓`/`OK` plus prose.
- Warning: existing `!`/`WARNING` plus prose.
- Error/destructive: existing `✗`/`ERROR` plus remediation.
- Do not depend on ANSI color; color is optional enhancement only.

## Copywriting Contract

| Element                  | Copy                                                                                                                                          |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Primary CTA              | Provision cloud runner                                                                                                                        |
| Empty state heading      | No RunnerKit-managed cloud runner is saved for `owner/name`.                                                                                  |
| Empty state body         | Run `runnerkit up --repo owner/name --cloud hetzner` to create one, or pass `--host user@host` to use an existing machine.                    |
| Error state              | Hetzner credentials are missing. Export `HCLOUD_TOKEN` or `HETZNER_CLOUD_TOKEN`, then rerun `runnerkit up --repo owner/name --cloud hetzner`. |
| Destructive confirmation | Destroy cloud runner: type `destroy owner/name` to remove the GitHub runner registration and RunnerKit-created Hetzner resources.             |

### Required prompt and output copy

| Surface                              | Required copy                                                                                                                                                                  |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Setup path prompt                    | Choose setup path for `owner/name`:                                                                                                                                            |
| BYO option                           | Use existing SSH host (BYO)                                                                                                                                                    |
| Cloud option                         | Provision recommended cloud runner (Hetzner)                                                                                                                                   |
| Missing non-interactive cloud intent | RunnerKit will not create billable cloud resources without an explicit `--cloud hetzner` flag and `--yes`.                                                                     |
| Provisioning plan heading            | Cloud runner provisioning plan                                                                                                                                                 |
| Cost caveat                          | Estimated cost is approximate. Provider pricing varies by region and time; billing stops only after RunnerKit-created billable resources are deleted or verified non-billable. |
| Provision confirmation               | Create billable Hetzner resources for `owner/name`?                                                                                                                            |
| Future cleanup line                  | Future cleanup: `runnerkit destroy --repo owner/name`                                                                                                                          |
| Readiness blocked                    | Cloud machine is not ready for runner registration yet. Fix the provider or SSH readiness issue, then rerun `runnerkit up --repo owner/name --cloud hetzner`.                  |
| Provider status line                 | Provider: Hetzner `fsn1` `cpx22` `ubuntu-24.04`, server `<id>`, state `<state>`, public host `<ip>`                                                                            |
| Destroy plan heading                 | Cloud destroy plan                                                                                                                                                             |
| Destroy billing warning              | This removes RunnerKit-created cloud resources that may still be billing. Local state is removed only after GitHub and provider cleanup are verified.                          |
| Partial destroy summary              | Cleanup incomplete. RunnerKit kept state with pending cleanup checkpoints so you can rerun `runnerkit destroy --repo owner/name`.                                              |
| Verified destroy summary             | Cloud runner destroyed. GitHub runner is absent and RunnerKit-created Hetzner resources are deleted or verified non-billable.                                                  |

Copywriting rules:

- CTA labels must be verb + noun, never `Submit`, `OK`, `Save`, or `Continue` alone.
- Every error must include a concrete next command, flag, env var, or provider action.
- Destructive copy must name both scopes: GitHub runner registration and RunnerKit-created cloud resources.
- Cost copy must always include the approximate/pricing-varies/user-responsible caveat before provisioning and in docs.
- Use `destroy` only for cloud billable resources; use `down` only for BYO cleanup.

## CLI Output Contract

### Provisioning plan fields

Human and JSON output must include these fields before cloud mutation:

- `provider`: `hetzner`
- `region`: `fsn1` unless user overrides it
- `server_type`: `cpx22`
- `image`: `ubuntu-24.04`
- `estimated_hourly_cost` and `estimated_monthly_cost` with `cost_estimate_caveat`
- `resources`: server, SSH key, firewall, public IPv4/IPv6, and explicit `no backups`, `no snapshots`, `no volumes`, `no floating IPs`
- `resource_names` and ownership tags including `runnerkit=true`, repo slug, runner name/state ID, mode, `managed=true`, and created-at
- `ssh_allowed_cidr`
- runner labels and exact `workflow_snippet`
- `future_destroy_command`: `runnerkit destroy --repo owner/name`

### Status/provider facts

Human `status`/`doctor` output must add a `Provider` source line alongside existing `State`, `GitHub`, `SSH`, `Service`, and `Labels` lines. JSON must add `sources.provider` with `kind`, `found`, `status`, `region`, `server_type`, `image`, `public_host`, `billable_resources`, `drift`, and `error` keys.

### Destroy result fields

Human and JSON output must include:

- artifact result for GitHub runner deletion or already absent
- remote runner/service cleanup result when SSH reachable
- provider server deletion result
- SSH key deletion result when RunnerKit-created
- firewall deletion result when RunnerKit-created
- Primary IP deletion/absence verification if tracked as billable
- local state removal result only after cleanup succeeds
- `partial_cleanup` boolean and `pending` checkpoint IDs when incomplete

## Documentation Contract

`docs/cloud-quickstart.md` and README cloud section must include these exact command examples:

```bash
export HCLOUD_TOKEN=...
runnerkit up --repo owner/name --cloud hetzner
runnerkit up --repo owner/name --cloud hetzner --yes
runnerkit status --repo owner/name
runnerkit logs --repo owner/name --since 30m --lines 200
runnerkit doctor --repo owner/name
runnerkit destroy --repo owner/name --dry-run
runnerkit destroy --repo owner/name
runnerkit destroy --repo owner/name --yes
```

The docs must include this exact workflow snippet example:

```yaml
runs-on: [self-hosted, runnerkit, runnerkit-owner-repo, linux, x64, persistent]
```

The docs must explicitly state:

- RunnerKit supports one recommended cloud path in Phase 4.
- The default cloud runner is persistent and intended for trusted private repositories.
- Ephemeral mode is deferred to Phase 5.
- RunnerKit prints labels/snippets and does not edit workflow YAML.
- Cost estimates are approximate and billing stops only after relevant provider resources are destroyed or verified non-billable.
