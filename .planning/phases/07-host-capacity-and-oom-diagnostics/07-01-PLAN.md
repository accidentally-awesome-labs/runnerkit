# Plan 07-01: Host capacity signals (Phase 1) + OOM / hard-kill diagnostics (Phase 2)

**Status:** Phase 1–2 implemented and released as **v1.0.9** (2026-05-12)  
**Goal:** Help RunnerKit users operating heavy or complex CI (e.g. large Rust test links, parallel native builds) avoid opaque “runner offline / shutdown signal” failures by (1) surfacing **RAM/swap headroom** early and in docs, and (2) when things go wrong, **classifying likely OOM / SIGKILL** from bounded host evidence with clear next steps.

**Out of scope for this plan:** GitHub Actions job API correlation, `.runnerkit.yml` repo hints, automatic workflow edits, or changing GitHub’s runner behavior.

---

## Background

- Preflight today validates disk (≥2 GiB), tools, time, network, sudo, etc. (`internal/preflight/checks.go`). It does **not** read memory or swap.
- Remote probe data lives in `remote.ProbeResult` (`internal/remote/executor.go`), populated by `SystemExecutor.Probe` (`internal/remote/system.go`) via discrete `ssh` invocations.
- `runnerkit doctor` already merges preflight into deep checks (`internal/ops/doctor.go`) and maps some findings to `RKD-*` codes (`internal/errcodes/codes.go`, `docs/troubleshooting/*.md`).
- `runnerkit logs` already collects bounded `journalctl` and `_diag` tails (`internal/ops/logs.go`).

## Phase 1 — RAM/swap preflight + troubleshooting doc

### 1.1 Extend remote probe with memory fields

- Add to `ProbeResult` (non-breaking for callers who ignore new fields):
  - `MemAvailableBytes int64` — parse `MemAvailable:` from `/proc/meminfo` (Linux standard).
  - `SwapFreeBytes int64` — parse `SwapFree:` from `/proc/meminfo` (0 if no swap).
- Implement in `SystemExecutor.Probe` with a **single** extra `ssh` round-trip where possible, e.g. one shell snippet that prints two integers (memavailable, swapfree) or two `grep` lines, with `0` on parse failure (do not fail the whole probe).
- **Constraints:** Linux-only path is OK (probe already assumes Linux for BYO). Document that cgroups / containers may make `MemAvailable` misleading (warning text, not failure).
- **Tests:** Unit-test parsing helpers with fixture strings; extend `remote` fake / tests if `ProbeResult` shape is asserted anywhere.

### 1.2 Preflight checks (warning-first)

- New check IDs in `internal/preflight/checks.go`, e.g.:
  - `host.mem_available` — **warning** (not failure) when `MemAvailable` is below a **documented** soft threshold (proposal: **4 GiB** = `4294967296` bytes) with remediation: “Heavy native link jobs (e.g. Rust integration tests) often need more RAM or reduced parallelism; see troubleshooting.”
  - `host.swap` — **warning** when `SwapFreeBytes == 0` and `MemAvailable` is below a second tier (proposal: **8 GiB**) — “No swap and limited RAM increases OOM risk during peak link.”
- **Do not** change `Report.Passed()` semantics for existing installs: new rows are **warnings** so `up` / register paths that require `Passed()` remain unchanged unless product explicitly gates on warnings later (defer).
- **Optional env override** (nice-to-have in Phase 1): `RUNNERKIT_PREFLIGHT_MEM_WARN_BYTES` — if set, use as warning threshold instead of default 4 GiB (helps CI matrix tests without magic numbers in every test).

### 1.3 Surface in `doctor` and JSON

- In `internal/ops/doctor.go`, map new preflight IDs to findings (severity mirrors preflight: warning → warning finding).
- Add `errcodes` entries + anchors under `docs/troubleshooting/bootstrap.md` (preflight / host capacity), e.g. `RKD-BOOT-0xx` for “low available memory” and “no swap + constrained RAM”.
- Update `docs/troubleshooting/README.md` component table if new codes are added.
- Ensure `--json` doctor payload includes enough structure that automation can see the new finding IDs (follow existing finding shape).

### 1.4 Documentation — OOM / link / runner death

- New file: `docs/troubleshooting/host-resources.md` (or a dedicated section in `bootstrap.md` if you prefer fewer files — **prefer new file** to keep bootstrap.md size manageable).
- Content (symptom → diagnosis → fix):
  - `collect2` / `ld` **signal 9**, Rust **test** binaries in **dev** profile, parallel links, GitHub “shutdown signal”, runner **offline** after heavy job.
  - Explain **why release build can pass** while **test** OOMs.
  - Mitigations: `CARGO_BUILD_JOBS=1` / `cargo test -j 1`, lighter `profile.test`, split jobs, larger VM, swap tradeoffs.
  - Point to **`runnerkit doctor`** / **`runnerkit logs`** after Phase 2 for evidence.
- Cross-link from `docs/troubleshooting/github.md` (runner offline) and/or `bootstrap.md` preflight area.

### Phase 1 verification

- `go test ./internal/preflight/... ./internal/remote/... ./internal/ops/... ./internal/cli/...` (or targeted packages touched).
- Golden / snapshot tests if doctor JSON is snapshotted elsewhere; otherwise assert finding IDs in doctor tests.
- Manual: run preflight against a small VM and confirm warning appears once under threshold.

## Phase 2 — OOM / kill “hints” in `doctor` (bounded evidence)

### 2.1 Goals

- When the user investigates a sick runner, **do not** require manual `dmesg` spelunking to suspect OOM.
- Output must be **bounded**, **redacted** where applicable, and **honest** (“likely OOM” vs “confirmed” — kernel messages are heuristics).

### 2.2 Evidence sources (priority order)

1. **Runner service journal** — reuse patterns from `internal/ops/logs.go` (`journalctl -u <unit> --since … -n …`). Extend or add a **doctor-only** collection with a default `--since` window (proposal: **24h** or **48h**, configurable flag `--since` on doctor if not already present — align with `logs` defaults).
2. **Kernel journal** — `journalctl -k -n 80 --no-pager` (often readable without `sudo`; prefer over `dmesg` when it works). If exit non-zero or empty, **degrade gracefully** (finding: “kernel log snippet unavailable; try sudo or SSH as root-capable user” as informational, not error).
3. **Optional:** grep runner journal for `SIGKILL`, `signal 9`, `OutOfMemory`, `oom-kill`, `Killed process`, `Runner listener exit` / GitHub runner strings — maintain a **small allowlisted regex set** in Go (not shell-injected user patterns).

### 2.3 Implementation shape

- New internal package or file, e.g. `internal/ops/hostkillhint.go`:
  - `type HostKillHint struct { Level string; Summary string; Evidence []string }` (or reuse existing finding structs).
  - Pure function `AnalyzeJournalForOOMHints(runnerJournal, kernelJournal string) []HostKillHint` for table-driven tests.
- `doctor` command path: after existing checks, if **SSH reachable** and **service failed or GitHub offline** (or a new `--deep` flag to always run — **recommend gated** to avoid noise on healthy runners), run **one** remote compound script with distinct `remote.Command` IDs, e.g. `doctor.journal.runner`, `doctor.journal.kernel`, each with **≤15s** timeout, max lines clamped (match `logs` clamp philosophy).
- Map strongest hint to a new **`RKD-*`** code (proposal: `RKD-BOOT-0xx` **or** `RKD-GH-0xx` if framed as “runner disappeared during work” — pick one component and stay consistent; **bootstrap** is reasonable because it is host-side).

### 2.4 CLI / UX

- Human `doctor`: extra finding block **“Host incident hints”** with 1–3 bullets (summary + “see RKD-…”) — no wall of log text by default.
- Optional `--with-log-snippets` flag: print **redacted** last N lines that matched (cap N ≤ 20) — reuse redaction pipeline from logs collection.
- JSON: include structured `host_incident_hints: [{ "id", "severity", "patterns_matched", "since" }]` (exact schema in implementation; keep backward compatible additive fields).

### 2.5 Safety and portability

- **No** broad `journalctl` without `-u` or `-k` and line caps.
- **cgroup / VPS** hosts: hint text should say memory pressure may be **container**-scoped if applicable (detect if `/proc/1/cgroup` suggests container — optional Phase 2 stretch; OK to skip v1 and mention in doc only).
- If `journalctl` missing (rare): skip kernel hint with informational finding.

### Phase 2 verification

- Unit tests: `AnalyzeJournalForOOMHints` with fixtures containing OOM killer lines, `ld` killed, runner shutdown, and negative cases (clean logs → no hints).
- Integration-style: fake `RemoteExecutor` scripted stdout for `doctor.*` command IDs; assert doctor emits expected finding IDs.
- Manual: reproduce or inject log fixture on a dev host; confirm human and JSON output.

## Rollout order

1. Land **Phase 1** first (probe + preflight + doctor mapping + docs + errcodes). Low risk, immediate value for new installs.
2. Land **Phase 2** second (hint analyzer + doctor remote collection + tests + docs cross-links). Touches remote execution paths — extra care on timeouts and redaction.

---

## Risks and mitigations

| Risk | Mitigation |
|------|------------|
| `MemAvailable` misleading in LXC | Warning copy + doc; never fail `Passed()` on memory alone |
| `journalctl -k` permission denied | Graceful skip + doc alternative (`sudo dmesg` manual) |
| False-positive OOM hints | Conservative regex set; label as “likely” not “confirmed” |
| Extra SSH latency on `doctor` | Gate heavy collection on failure/offline or `--deep` |
