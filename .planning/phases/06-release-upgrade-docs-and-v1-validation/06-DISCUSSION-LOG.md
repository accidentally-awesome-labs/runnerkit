## Distribution + install

### Q1. Which install channels should v1 ship through? (multiSelect)

| Option | Description | Selected |
|--------|-------------|----------|
| GitHub Releases binaries | Pre-built binaries per platform/arch attached to git tags. Always supported regardless of other channels. | ✓ |
| Homebrew tap | `brew install runnerkit` on macOS/Linux. Fast UX for solo devs. Requires maintaining a tap repo + formula updates per release. | ✓ |
| go install | `go install github.com/.../cmd/runnerkit@latest`. Free for Go-tooled devs, bypasses signing/checksums. | |
| Linux .deb / .rpm | Native package install for Debian/Ubuntu/RHEL. Extra packaging surface and repo hosting cost. | |

**User's choice:** GitHub Releases binaries + Homebrew tap.

### Q2. Which CLI host platforms must v1 cover? (multiSelect)

| Option | Description | Selected |
|--------|-------------|----------|
| macOS arm64 + amd64 | Solo dev's daily driver. Apple Silicon + Intel Macs. | ✓ |
| Linux amd64 + arm64 | Server/dev Linux. amd64 for most VMs, arm64 for Hetzner cax/Ampere/Pi-class boxes. | ✓ |
| Windows amd64 | Run CLI from Windows desktop, target Linux runner via SSH. Adds CI matrix and `\` path edge cases. | |
| Linux 386 / armv6/v7 | Older 32-bit boxes. Runner pin (2.334.0) doesn't target these. | |

**User's choice:** macOS arm64+amd64, Linux amd64+arm64.

### Q3. How should release builds be produced? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| GoReleaser via GitHub Actions | Standard for Go CLIs. One config produces binaries + checksums + Homebrew formula + Release notes from a tag push. | ✓ |
| Hand-rolled GitHub Actions matrix | Custom workflow with per-platform `go build` + manual upload. More control, more maintenance. | |
| Local manual cuts | `make release` on maintainer's machine. Simplest now, ties releases to one machine. | |

**User's choice:** GoReleaser via GitHub Actions.

### Q4. What integrity artifacts must each release include? (multiSelect)

| Option | Description | Selected |
|--------|-------------|----------|
| SHA256 checksums file | `runnerkit_v1.0.0_checksums.txt` covering every binary. Minimum download-integrity bar. | ✓ |
| Cosign keyless signature | Sigstore keyless signing via GitHub Actions OIDC. No key custody. | ✓ |
| GPG signature | Detached `.asc` signature. Requires maintainer key management + distribution. | |
| SBOM (CycloneDX or SPDX) | Bill of materials per release. Useful for supply-chain audits, generated by GoReleaser. | |

**User's choice:** SHA256 checksums + cosign keyless.
**Notes:** GPG and SBOM noted as deferred — revisit when downstream packager/auditor demand appears.

## Upgrade + runner rot

### Q1. How should the CLI surface that a new RunnerKit version is available? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Lazy check on relevant commands | HEAD against GitHub Releases on `up`/`status`/`doctor`, cached ~24h, skipped offline/JSON. One-line non-blocking notice. | ✓ |
| Manual `runnerkit upgrade-check` only | Zero unsolicited network traffic. User runs the command to learn about updates. | |
| Always-on check on every command | Hits GitHub on every CLI invocation. Most surface, most noise, most failure modes when offline. | |
| No update check, ever | CLI never reaches out. Undermines REL-05's anti-rot promise. | |

**User's choice:** Lazy check on relevant commands.

### Q2. How should developers actually install a new RunnerKit version? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Delegate to the install channel | `runnerkit upgrade` prints `brew upgrade runnerkit` or `Download from <release URL>` based on detected install path. | ✓ |
| Self-update binary in place | `runnerkit upgrade` downloads + replaces its own binary. Adds significant code (download, verify, atomic replace, rollback). | |
| No `runnerkit upgrade` command | Document upgrade in README/docs only. Smaller CLI surface. | |

**User's choice:** Delegate to install channel.

### Q3. How should the pinned GitHub Actions runner version (currently 2.334.0) be managed? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Version pin per RunnerKit release + `runnerkit upgrade-runner` | Each release pins a known-good runner version. Doctor warns when stale; `upgrade-runner` re-runs bootstrap. | ✓ |
| Always fetch latest on install + warn-only on rot | Bootstrap pulls latest at install time. Doctor warns on deprecation. No bundled `upgrade-runner`. | |
| Stay pinned forever, doctor warns only | Don't change the pin in v1. Just warn when GitHub eventually rejects it. | |

**User's choice:** Version pin per release + `runnerkit upgrade-runner`.

### Q4. What should happen when RunnerKit reads state with a newer or older `schema_version`? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Forward-only auto-migrate, block on unknown | Old state -> auto-migrate forward (with backup). Newer state -> refuse to mutate, print upgrade guidance. | ✓ |
| Always prompt before migration | Even forward-compatible old state requires user confirmation. | |
| Free-form best-effort migrate | Try to read whatever's there; coerce missing fields. High risk of silent drift. | |

**User's choice:** Forward-only auto-migrate, block on unknown.

## Live v1 validation

### Q1. What should plan 06-04's end-to-end v1 validation actually exercise? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Live BYO + live Hetzner + 10-min stopwatch checklist | Both pending live smokes + documented stopwatch checklist a fresh user can run on a clean machine. | ✓ |
| Stopwatch checklist only | Doc the exact happy-path commands; tester runs on a clean machine and records durations. Defers live-smoke debt. | |
| Live cloud smoke only, no stopwatch | Validates infrastructure but doesn't directly prove the 10-minute promise. | |

**User's choice:** Live BYO + live Hetzner + 10-min stopwatch checklist.

### Q2. How should live smokes be triggered? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Manual `make smoke-live` / docs procedure | Maintainer runs locally with secrets set, before tagging a release. No CI workflow, no cost surprises. | ✓ |
| Manual-dispatch GitHub Actions workflow | `workflow_dispatch` workflow with required secrets. Repeatable, but stores real Hetzner + GitHub PAT in repo secrets. | |
| Scheduled / pre-release CI gate | Auto-run nightly or on `v*` tag. Strongest signal, burns cloud cost, most fragile. | |

**User's choice:** Manual `make smoke-live`.

### Q3. How should the Hetzner live smoke contain billable risk? (multiSelect)

| Option | Description | Selected |
|--------|-------------|----------|
| Hard pre-check that the configured Hetzner project is empty/scoped | Smoke verifies target Hetzner project has zero pre-existing managed resources with the runnerkit prefix; refuses to run otherwise. | ✓ |
| Deferred destroy verification with timeout | After smoke, force `runnerkit destroy --yes` and assert provider returns 404 for every created ID within N minutes. | ✓ |
| Cheapest profile pin (cax11 or cpx11) | Override default `cpx22` with cheapest viable Hetzner profile for smoke runs. | |
| No special guards | Trust Phase 4's destroy semantics + manual cleanup. Lowest setup, biggest blast radius. | |

**User's choice:** Empty-project pre-check + deferred destroy verification with timeout.
**Notes:** Profile pin not selected — implies smoke runs against the default `cpx22` so the validated path matches what real users hit.

### Q4. Where should validation results live? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| 06-VERIFICATION.md + RELEASE-NOTES per release | Phase 6's VERIFICATION captures v1.0.0 baseline; each release ships `RELEASE-NOTES-vX.Y.Z.md` with checklist + measured duration. | ✓ |
| VERIFICATION.md only | Capture baseline once for v1.0.0; future releases re-run but don't formally publish results. | |
| Private maintainer notes only | Don't publish durations or smoke results. | |

**User's choice:** 06-VERIFICATION.md + per-release RELEASE-NOTES.

## Troubleshooting doc shape

### Q1. How should DOC-04's troubleshooting docs be organized on disk? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Split per-component under docs/troubleshooting/ | `auth.md`, `ssh.md`, `bootstrap.md`, `github.md`, `provider.md`, `cleanup.md`, plus index. Easier to grow, easier to deep-link. | ✓ |
| One docs/troubleshooting.md | Single file with H2 sections per component. Simpler now, harder to anchor-link. | |
| Recovery recipes inside existing docs | Extend README + quickstarts + safety with troubleshooting subsections. Lowest doc-tree churn, nothing to deep-link to. | |

**User's choice:** Split per-component under `docs/troubleshooting/`.

### Q2. How should `runnerkit doctor` and error output point users at the right doc? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Stable error codes + URLs | `RKD-AUTH-001` etc. CLI prints `See: <docs URL>/troubleshooting/<component>#rkd-<component>-NNN`. | ✓ |
| Topic anchors only | CLI prints `See: docs/troubleshooting/auth.md` without per-finding codes. | |
| No deep links | CLI mentions `runnerkit doctor` and the docs site by name; user navigates. | |

**User's choice:** Stable error codes + URLs.

### Q3. What failure surface should the v1 docs cover? (multiSelect)

| Option | Description | Selected |
|--------|-------------|----------|
| Setup failures (auth scope, public-repo block, SSH host-key, preflight) | Phase 1–2 surface. Most common point of first-encounter failure. | ✓ |
| Bootstrap + service failures (systemd unit, runner user, package install, online verification timeout) | Phase 2 surface. Real failures after successful auth/SSH. | ✓ |
| Operations failures (status drift, doctor findings, recover, down) | Phase 3 surface. Hardest for users to self-diagnose without docs. | ✓ |
| Cloud + cleanup failures (Hetzner quota/credentials, partial destroy, billable-resource verification) | Phase 4–5 surface. Cost-sensitive, can't be skipped for v1. | ✓ |

**User's choice:** All four — full failure surface from Phases 1–5.

### Q4. What style should each troubleshooting entry use? (single)

| Option | Description | Selected |
|--------|-------------|----------|
| Symptom → Diagnosis → Fix (with copyable commands) | Short structured entries optimized for the user already stuck. | ✓ |
| Long-form prose explanations | More tutorial-style, fewer copy-paste recipes. | |
| FAQ Q→A list | Single flat list of `Q: ... A: ...`. Fast to scan, harder to organize cross-component. | |

**User's choice:** Symptom → Diagnosis → Fix.
